Negative Examples Should Not Get More Timeline Than Positive Ones

Sampling synthetic decision cut-points so non-events use realistic partial histories

View Repository Explore Method

Overview

This project predicts escalation risk in enterprise customer support cases using metadata, event history, milestones, and comments.

The hardest part was not the model. It was defining how much of each case timeline the model was allowed to see.

Escalated cases naturally stop at the escalation decision point. Non-escalated cases do not. If full timelines are kept for non-events, the negative class receives more future information than the positive class.

Architecture

The Core Idea

Event-based ML is not only about defining the label. It is also about defining the decision moment.

Positive cases were observed only up to escalation start. To make the non-event class comparable, synthetic decision cut-points were sampled from the empirical distribution of real escalation timing.

Negative cases should not get a longer timeline than escalated ones.

Selected code snippet

escalation_points_dist <- quantile(
    escalation_starts[, escalation_start / case_end],
    seq(0, 1, 0.01)
)

other_cases[, time_cut := case_end * sample(escalation_points_dist, .N, replace = TRUE)]

Why this mattered

  • removed unfair history-length asymmetry between classes
  • made training examples resemble real decision snapshots
  • reduced look-ahead advantage in non-event cases
  • improved the realism of label and feature construction together

Where this pattern generalizes

  • churn and retention event modeling
  • support escalation prediction
  • failure and incident anticipation
  • any event process with an ambiguous negative-class decision moment

Key Takeaway

The non-event class also needs a realistic decision horizon.

Built as a public project showcase for realistic observation-window design in event-based ML.
